# 妖币雷达待完善项与开发路线图

> 归档说明：本文档形成于 `Phase 2` 阶段，主要用于保留当时的问题清单与演进路径，不代表当前 `Phase 3.x` 实现状态。
> 当前实现请优先参考 [SKILL.md](/Users/zhangpeng/opt/meme-coin-radar/skills/meme-coin-radar/SKILL.md) 与 `scripts/` 下实际代码。

> 更新时间：2026-04-25
> 目标：梳理当前“获取数据 / 分析数据 / 评估妖币”流程中仍未完善的部分，并给出可直接分派给其他 agent 的改造建议。

---

## 一句话结论

当前项目已经具备 **Phase 2 可运行能力**，但还不能算“完善”。

最强的部分：
- 评分框架已经成型
- 多源数据已经接通
- 报告与交易计划已经有可用雏形

最弱的部分：
- 候选发现仍偏“抽样扫描”，不是全市场发现
- 数据质量与失败处理不够健壮
- 历史分析能力不足，很多判断还是单次快照

建议总体方向：
- **P0** 先补数据健壮性和候选池完整性
- **P1** 再补历史分析和实体映射
- **P2** 最后补回测、监控、自动化决策增强

## 二、获取数据：仍需完善的部分

### 1. 数据获取失败时缺少结构化错误信息

现状：
- `scripts/providers/common.py` 中 `run()` / `json_out()` / `http_json()` 失败后大多直接返回 `None`
- 上层只能知道“没数据”，不知道是命令不存在、超时、网络失败、解析失败，还是权限问题

影响：
- 难排查生产问题
- 难做监控与告警
- 评分阶段无法区分“数据真的缺失”还是“本次拉取异常”

建议改造：
- 为 provider 层增加统一返回结构，例如：
  - `{"ok": True, "data": ...}`
  - `{"ok": False, "error_type": "timeout|network|parse|command", "message": "..."}`
- 在 `batch_binance()`、`okx_btc_status()`、`okx_swap_tickers()` 中保留原始失败原因
- 在输出目录中新增 `fetch_status.json`

验收标准：
- 任一 provider 失败时，报告中可以看到失败类型
- `result.json` 或单独状态文件中可统计每个数据源成功率

优先级：`P0`

---

### 2. OI 数据没有 fallback

现状：
- `scripts/providers/binance.py` 中 `open_interest()` 只调用 Binance HTTP API
- 一旦 Binance OI 请求失败，评分阶段直接缺核心字段 `oi`

影响：
- M3 评分退化
- 容易触发人工复核或降级
- 可执行信号稳定性下降

建议改造：
- 优先补一个 fallback 层：
  - 如果 Binance OI 拉取失败，至少记录显式状态，而不是静默空值
  - 评估可否从 Hyperliquid 或其他可用 perp 数据源补 OI 或 OI proxy
- 若短期没有替代数据源，也要把 OI 缺失标为“数据源失败”，而非“币种天然缺字段”

验收标准：
- OI 缺失时，输出中明确区分：
  - `source_unavailable`
  - `field_not_supported`

优先级：`P0`

---

### 3. 候选池不是全市场发现，容易漏妖币

现状：
- 候选池来自三部分拼接：
  - `key_coins`
  - OKX 全量 tickers 的涨跌幅前后 20
  - Binance Alpha 活跃币前 15
- 这意味着当前更像“重点池扫描”，而不是全市场发现

影响：
- 可能错过未进入 `key_coins`、也未在 OKX 异动前列、但链上已经很热的新妖币
- GMGN 热门币目前多数只进入 Layer 0 观察区，没有进入主评分流程

建议改造：
- 增加“候选发现层”，将候选来源分成：
  - CEX 异动候选
  - Alpha 热度候选
  - GMGN 热门候选
  - GMGN 聪明钱候选
  - 新币候选
- 给每个候选打上 `candidate_source`
- 允许链上候选进入主评估，但区分：
  - `tradable_on_cex = true/false`
  - `market_type = cex_perp / onchain_spot / layer0_watch`

验收标准：
- 同一份扫描结果中，可以同时看到：
  - 可交易合约候选
  - 纯链上热点候选
- 两类候选共享主评分框架，但输出状态不同

优先级：`P0`

---

### 4. 数据新鲜度没有显式管理

现状：
- 每次扫描都是即时拉取，但没有记录各字段时间戳
- 同一份报告内，BTC、ticker、funding、GMGN、OI 的采样时间可能不一致

影响：
- 在快节奏行情里，混合旧数据和新数据会让评分失真
- 后续回看时难复盘

建议改造：
- 所有 provider 返回值增加：
  - `fetched_at`
  - `source`
  - `latency_ms`（可选）
- 在报告和 JSON 中增加数据时间信息
- 超过阈值的旧数据标记为 stale

验收标准：
- 报告中可以看到关键模块数据时间
- 评分阶段可识别 stale data 并降低权重或标记复核

优先级：`P1`

## 三、分析数据：仍需完善的部分

### 5. 分析仍然是单次快照，缺少历史比较

现状：
- 当前分析依赖当次拉取的 1H / 4H / 1D K 线
- 代码里已经预留 `atr_vs_30d`，但尚未实现

影响：
- 无法判断“今天的波动是不是异常强”
- 无法做真正的相对强弱和持续性分析

建议改造：
- 建立本地历史快照存储：
  - ticker 快照
  - funding 快照
  - OI 快照
  - GMGN 热门与信号快照
- 基于历史数据补充：
  - `atr_vs_30d`
  - `volume_vs_7d`
  - `oi_vs_7d`
  - `alpha_count24h_vs_7d_avg`

验收标准：
- 评分不再只用“绝对值”，而是同时参考“相对过去 N 天偏离度”

优先级：`P1`

---

### 6. 链上分析深度还不够

现状：
- 现在主要使用 GMGN 提供的聚合字段：
  - `smart_degen_count`
  - `renowned_count`
  - `holders_growth_24h`
  - `rug_ratio`
  - `top_10_holder_rate`
- 但没有更细的链上行为拆分

影响：
- 更像“链上摘要评分”，而不是深度链上分析
- 对真实妖币的启动节奏、接力质量、持仓结构判断还不够精细

建议改造：
- 新增链上特征：
  - 新钱包占比
  - 巨鲸净买入/净卖出
  - 持仓簇集中度
  - 连续信号次数
  - 信号后市值扩张速度
- 若 GMGN API 不足，可预留其他链上源接入位

验收标准：
- M3 不再主要依赖少数字段
- 可以区分“真聪明钱接力”与“短暂刷量/喊单”

优先级：`P1`

---

### 7. 技术结构分析还缺盘口和成交行为维度

现状：
- 当前技术分析主要是 ATR、EMA、chg、volume、4H 共振

影响：
- 对极端短线妖币来说，单看 K 线和成交额不够
- 缺少对“爆发前吸筹 / 爆发后派发”的结构判断

建议改造：
- 增加可选分析因子：
  - 连续阳线/阴线数量
  - 上下影比例
  - 实体占比
  - 突破后回踩结构
  - 盘口深度和滑点估算（若数据源支持）

验收标准：
- M2 能更好区分“强趋势启动”与“末端拉升”

优先级：`P2`

---

## 四、评估妖币：仍需完善的部分

### 8. “妖币发现”与“可交易评估”仍然混在一起

现状：
- 当前系统主要面向 Binance/OKX 可交易合约
- GMGN 热门币更多是附加观察板块，而不是评估主干

影响：
- 对“妖币”这个概念的覆盖不完整
- 真正的链上妖币可能被看到，但不会进入同等级评估

建议改造：
- 将评估结果拆成三类：
  - `tradable_candidate`
  - `watch_only_candidate`
  - `rejected_candidate`
- 增加统一输出字段：
  - `tradability`
  - `venue`
  - `reason_not_tradable`

验收标准：
- 报告里明确区分“这是不是妖币”和“这能不能做”
- 纯链上妖币也能有完整评分，只是不会进入合约执行区

优先级：`P0`

---

### 9. GMGN 与 CEX 标的映射方式不够稳

现状：
- 当前通过 symbol 做映射
- `FARTCOIN` 这类映射可以工作，但同名币、跨链同 symbol、假冒 symbol 容易出错

影响：
- 链上信息可能挂错币
- 安全评分和聪明钱评分可能污染合约评估

建议改造：
- 建立统一资产映射层：
  - `symbol`
  - `base_symbol`
  - `chain`
  - `contract_address`
  - `cex_symbol`
  - `mapping_confidence`
- 对低置信映射禁止直接并入主评分

验收标准：
- 每个 GMGN -> CEX 映射都有置信度
- 低置信映射仅做观察，不直接进入执行建议

优先级：`P0`

---

### 10. 缺失字段降级机制仍较粗糙

现状：
- 当前只要命中核心字段缺失，就进入 `needs_manual_review`
- 核心字段缺失 3 项以上，统一降级到 `watchlist`

影响：
- 逻辑简单，但还不够细
- 无法区分“字段重要但暂时失败”与“该资产天然不存在该字段”

建议改造：
- 增加字段缺失分类：
  - `missing_due_to_fetch_error`
  - `missing_due_to_not_supported`
  - `missing_due_to_asset_type`
- 按字段重要性定义降权，而不是只有硬降级

验收标准：
- `needs_manual_review` 的原因更加具体
- 不同缺失原因导致不同评分处理

优先级：`P1`

---

### 11. 缺少回测验证，评分可信度还未闭环

现状：
- 当前评分规则看起来合理，但没有系统回测

影响：
- 无法量化：
  - 75 分以上的胜率
  - 不同模块对收益率的贡献
  - 硬否决规则的实际效果

建议改造：
- 建立简单回测框架：
  - 输入：历史扫描快照
  - 输出：后续 1h / 4h / 24h 收益、最大回撤、胜率
- 对评分阈值做回测优化

验收标准：
- 可以回答“75 分以上到底值不值得做”
- 可以比较不同版本模型优劣

优先级：`P2`

## 五、推荐开发优先级

### P0：必须先补

1. provider 错误结构化
2. OI fallback / OI 缺失显式标注
3. 候选发现层重构，让链上热点进入主评分
4. 可交易性与妖币性拆分
5. GMGN 与 CEX 资产映射层

### P1：建议第二阶段补

1. 数据新鲜度与时间戳
2. 历史快照存储
3. 相对强弱指标：ATR 30d、volume 7d、alpha 7d
4. 链上深度特征补强
5. 缺失字段分类降级

### P2：增强项

1. K 线微结构与盘口因子
2. 回测框架
3. 模型参数优化
4. 监控与告警面板

## 七、推荐验收顺序

1. 先验收数据层是否稳定
2. 再验收候选池是否不再漏掉链上热点
3. 再验收资产映射是否可靠
4. 最后验收历史因子与回测

建议里程碑：
- `Phase 2.1`：数据健壮性 + 候选发现重构
- `Phase 2.2`：资产映射 + 历史快照
- `Phase 3.0`：回测闭环 + 模型优化

---

## 八、最终建议

如果现在就继续开发，最值得优先推进的不是“再加新指标”，而是先把以下两件事做好：

1. **不要再漏妖币**：让链上热门和聪明钱信号进入主候选池
2. **不要误判妖币**：让数据失败、映射错误、字段缺失不再静默污染评分

把这两件事补好之后，这个项目才会从“能跑的评分器”升级成“更可靠的妖币雷达”。
